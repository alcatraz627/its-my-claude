## Brief

Scans the project and generates a comprehensive index of directory structure, key files, dependencies, and architectural patterns — saved as a markdown file and rendered as an HTML report. Runs in an isolated forked context so hundreds of file reads don't pollute the main conversation.

# Project Index Builder Skill

Builds a comprehensive index of the project's structure, identifying key files, components, dependencies, and architectural patterns. The index can be saved as a markdown file for documentation or used to answer questions about the codebase.

**🔄 Self-Updating Skill:** This skill adapts its own documentation based on what it finds in your project. After running, it will update its examples and patterns to match your actual tech stack, removing irrelevant examples and adding new categories for patterns it discovers.

## Usage

```
/project-index [latest | output_path]
```

**Arguments:**

- `latest` — Skip scanning entirely. Regenerate the HTML report from the last saved index and open it in Chrome. If no previous index exists, report that and ask the user if a new scan should be run.
- `output_path` (optional): Path where the index should be saved (e.g., `docs/PROJECT_INDEX.md`). If not provided, defaults to `.claude/project-index.md`

**Persistent files maintained by this skill:**

- `.claude/project-index.md` — The generated index (overwritten each run)
- `.claude/indexing-notes.md` — Append-only run log; each entry is dated and summarizes what changed
- `.claude/output/<datetime>-project-index/` — Timestamped HTML report folder generated by `/create-report` (a new folder is created each run; previous ones are not deleted)

## What This Skill Indexes

### 1. Project Overview

- Repository name and location
- Main technology stack (framework, language, package manager)
- Project type (frontend, backend, fullstack, monorepo)
- Key configuration files

### 2. Directory Structure

- Top-level directories and their purposes
- Important subdirectories
- Naming conventions and patterns

### 3. Key Files Inventory

- **Configuration files**: package.json, tsconfig.json, next.config.js, etc.
- **Entry points**: main app files, server files, index files
- **Routing**: route definitions, API endpoints
- **State management**: stores, contexts, reducers
- **Types/Interfaces**: shared type definitions
- **Utilities**: helper functions, common utilities
- **Components**: UI component structure

### 4. Dependencies Analysis

- Production dependencies (from package.json)
- Dev dependencies
- Key libraries and their purposes
- Internal dependencies (workspace packages)

### 5. Architectural Patterns

- Design patterns in use (HOC, render props, hooks, etc.)
- Code organization patterns
- API communication patterns
- State management approach
- Styling approach (CSS modules, Tailwind, styled-components, etc.)

**Package URL Resolution:**

For every dependency listed in the index, resolve its canonical repository URL and write it as a markdown link `[package-name](url)`. Follow these rules:

1. **npm packages**: URL is `https://www.npmjs.com/package/<name>`. For scoped packages like `@tanstack/react-query`, the URL is `https://www.npmjs.com/package/%40tanstack%2Freact-query` (encode `@` as `%40`, `/` as `%2F`) — OR simply use `https://npmjs.com/package/@tanstack/react-query` (npmjs.com accepts unencoded scoped names).
2. **Well-known GitHub-first packages**: If you know the canonical GitHub repo (e.g., `drizzle-orm` → `https://github.com/drizzle-team/drizzle-orm`), prefer the GitHub URL over npmjs.
3. **Format**: Write each dependency as `[package-name](url) (version) — purpose description`

Example output in the markdown:

```
- [react](https://npmjs.com/package/react) (^19.2.3) — UI library
- [drizzle-orm](https://github.com/drizzle-team/drizzle-orm) (^0.39.1) — PostgreSQL ORM
- [@tanstack/react-query](https://npmjs.com/package/@tanstack/react-query) (^5.x) — Server state management
```

### Step 5.5: Document Key Utility Exports

The index currently stops at the **file level** — it lists `use-q.ts` but not `useQ`, lists
`with-server.ts` but not `withServer`. This step fills that gap by scanning for named exports
across all utility modules and building a symbol-level reference table.

**Target files to scan** (use Grep, not Bash grep):

```
src/utils/*.ts            — top-level utility modules
src/utils/hooks/*.ts      — all custom hook files
src/core/context/*.ts     — context factories (createKeyedContext)
src/core/auth/*.ts        — auth utilities (useUser, etc.)
```

**Grep pattern** — finds every top-level named export:

```
pattern: "^export (function|const|class|type|interface|enum)"
```

Run this across each target directory. For each match, extract:

- **Symbol name** — the identifier immediately after `function`/`const`/`class`/`type`/`interface`/`enum`
- **File** — relative path (strip `src/` prefix for brevity)
- **Description** — infer from the name, or read the first JSDoc comment above it if one exists

**Filter rules — include:**

- Exported functions and hooks with meaningful names (e.g., `useQ`, `withServer`, `QueryKeys`)
- Exported enums and constants that act as reference tables (e.g., `Routes`, `QueryParams`, `StorageKey`)
- Exported factory functions (e.g., `createKeyedContext`, `getQueryClient`)
- Exported type aliases that are used broadly (e.g., `LoadingState`, `Func`, `StorageKey`)

**Filter rules — skip:**

- Internal helper types that are only consumed in the same file
- Re-exports (`export { x } from "./y"`) — these belong in the source file's entry
- Trivially named exports (`export const x = 1`)
- Generated or boilerplate exports (e.g., `export {}` barrel stubs)

**Output format** — a markdown table in a `## Key Utility Exports` section:

```markdown
## Key Utility Exports

| Symbol               | File                              | Description                                                   |
| -------------------- | --------------------------------- | ------------------------------------------------------------- |
| `useQ`               | `utils/hooks/use-q.ts`            | Shorthand `useQuery` wrapper with typed args                  |
| `useM`               | `utils/hooks/use-m.ts`            | Shorthand `useMutation` wrapper with toast feedback           |
| `withServer`         | `utils/with-server.ts`            | Marks a module as server-only; throws if imported client-side |
| `QueryKeys`          | `utils/react-query.ts`            | Typed factory object for all TanStack Query cache keys        |
| `Routes`             | `utils/routing.ts`                | Enum of all app route path constants                          |
| `QueryParams`        | `utils/url.ts`                    | Enum of URL search parameter names                            |
| `createKeyedContext` | `core/context/global-context.tsx` | Generic typed React Context factory                           |
| `getQueryClient`     | `utils/react-query.ts`            | Singleton TanStack Query client accessor                      |
```

**Prioritization**: scan `src/utils/hooks/` first (hooks are the most-called exports), then
`src/utils/*.ts` top-level modules, then `src/core/context/`. Aim for 15–35 rows — enough to
be useful as a lookup table without listing every internal helper.
